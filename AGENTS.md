# AGENTS.md

## Назначение

Этот документ задаёт единые правила для всех автоматизированных и ручных изменений в репозитории `usb-driver-fuzzing-thesis`.

Контекст проекта: выпускная квалификационная работа по исследованию устойчивости USB-драйверов и компонентов USB-стека ОС к аномальному поведению USB-устройства.

## Обязательные инварианты

1. Целевой стенд USB-экспериментов: **RP2040 / Waveshare RP2040 Zero**.
2. Основные языки реализации: **C** и **Python**.
3. Базовая среда: **WSL Ubuntu + Docker Desktop + Pico SDK**.
4. Текущий baseline `portable_demo` должен оставаться **собираемым и работоспособным**.
5. Нельзя ломать существующий quick-start из `README.md` и текущие build/flash-скрипты в `tools/`.

## Границы безопасности и этики

В рамках данного репозитория и ВКР запрещено:

1. Реализовывать вредоносные BadUSB-полезные нагрузки.
2. Реализовывать reverse shell, создание административных пользователей, кражу учетных данных, persistence-механизмы.
3. Реализовывать автоматическое выполнение команд на хосте.

Разрешённый практический фокус:

1. Безопасные эксперименты уровня USB descriptor/enumeration.
2. Вариации дескрипторов: device/configuration/interface/endpoint/string.
3. Базовые устройства CDC/HID для контролируемых тестов.
4. Наблюдение и анализ OS-логов, USB-трейсов и реакций драйверного стека.

## Политика изменений кода

1. Предпочитать малые, ревью-пригодные изменения и коммиты.
2. Любые новые функции должны быть изолированы от baseline `portable_demo`.
3. Для новых USB-кейсов допускается отдельный firmware-target, не влияющий на сборку `portable_demo`.
4. Не выполнять рефакторинг `nRF24`/`RFV2`/control-plane кода без строгой необходимости для текущей задачи.
5. При необходимости изменений в существующих модулях фиксировать мотивацию в описании PR/коммита.

## Политика Python-инструментов

1. Предпочтение отдаётся стандартной библиотеке Python.
2. Внешние зависимости добавляются только при явном техническом обосновании (воспроизводимость, отсутствие адекватной stdlib-альтернативы, существенное снижение сложности).
3. Для каждой новой зависимости требуется краткое обоснование в документации.

## Документация

1. Документация ведётся на русском языке.
2. Стиль: академический, технически точный, воспроизводимый.
3. Для каждого эксперимента должны быть зафиксированы:
   - цель и гипотеза;
   - параметры USB-кейса;
   - платформа и версия ОС;
   - порядок запуска;
   - наблюдения (логи/трейсы);
   - итог и интерпретация.

## Ожидаемое направление развития

Необходимо развивать отдельный **experiment layer** с воспроизводимой структурой:

1. Каталог кейсов с формализованными описаниями сценариев.
2. Шаблоны результатов (таблицы/формы наблюдений).
3. Скрипты для повторяемого сбора артефактов.
4. Методика наблюдений отдельно для Windows и Linux.

Рекомендуемая минимальная схема (может уточняться):

```text
experiments/
  cases/
  templates/
  scripts/
  methodology/
results/
  raw/
  processed/
```

## Критерии приемки изменений

Изменение считается корректным, если одновременно выполняются условия:

1. `portable_demo` продолжает собираться штатным способом.
2. Быстрый путь из `README.md` остаётся валидным без дополнительных ручных шагов.
3. Новая функциональность ограничена безопасными USB descriptor/enumeration экспериментами.
4. Добавленная документация достаточна для воспроизведения эксперимента другим исследователем.

## Приоритет при конфликте требований

При конфликте требований приоритет следующий:

1. Безопасность и этические ограничения.
2. Сохранение работоспособности baseline (`portable_demo`, build/flash-скрипты, quick-start).
3. Воспроизводимость экспериментов.
4. Локальные архитектурные улучшения.
