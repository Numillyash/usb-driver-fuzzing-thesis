# Методология экспериментов с USB descriptor/enumeration аномалиями

## 1. Методологическая цель

Обеспечить воспроизводимую процедуру выполнения и анализа безопасных USB-экспериментов на RP2040 с сопоставимыми результатами для Windows и Linux.

## 2. Общий принцип

Каждый эксперимент задаётся формальным JSON-кейсом. Кейс описывает базовый профиль устройства, конкретную мутацию дескриптора и ожидаемые реакции ОС. Выполнение кейса сопровождается сбором логов и USB-трейсов по стандартизированному шаблону.

## 3. Единица эксперимента

Единицей является один `case_id` из каталога `experiments/cases/`.

Обязательные поля кейса:
- `case_id`;
- `name`;
- `group`;
- `base_persona`;
- `description`;
- `mutation`;
- `expected_windows`;
- `expected_linux`;
- `risk_level`;
- `notes`.

## 4. Порядок выполнения

1. Зафиксировать параметры среды:
- модель RP2040-платы;
- версию прошивки экспериментального таргета;
- ОС, версию ядра/сборки и ключевые USB-драйверы;
- используемые средства трассировки.

2. Выполнить контрольный baseline-кейс соответствующей группы.

3. Выполнить целевой аномальный кейс.

4. Собрать артефакты:
- системные логи;
- USB-трейс перечисления;
- отметки времени подключения/отключения;
- итоговое состояние устройства в ОС.

5. Заполнить шаблон результата и выполнить первичную классификацию реакции.

## 5. Классификация наблюдаемых реакций

Рекомендуемые классы реакции:

1. `ACCEPTED_WITHOUT_WARNING`:
- устройство перечислено;
- явных предупреждений/ошибок не зафиксировано.

2. `ACCEPTED_WITH_WARNING`:
- устройство перечислено;
- присутствуют предупреждения, сообщения о неконсистентности или fallback-поведение.

3. `PARTIAL_ENUMERATION`:
- частичная инициализация;
- отдельные интерфейсы/endpoint недоступны.

4. `ENUMERATION_FAILED`:
- конфигурация не завершена;
- устройство не переходит в рабочее состояние.

5. `RE_ENUMERATION_LOOP`:
- повторяющиеся попытки инициализации/переподключения без стабильного результата.

6. `HOST_STACK_ANOMALY`:
- нетипичное поведение стека (существенные задержки, нестандартные ошибки, деградация подсистемы USB без выполнения вредоносной нагрузки).

## 6. Сопоставимость между ОС

Для сопоставления результатов необходимо:
- использовать идентичные case-файлы;
- сохранять единый формат итогового отчёта;
- фиксировать различия в реакциях по одинаковым классификационным критериям;
- отдельно отмечать особенности драйверных моделей Windows и Linux.

## 7. Ограничения интерпретации

- Результаты описывают поведение конкретных версий ОС и драйверов.
- Наблюдения не экстраполируются на все аппаратные конфигурации без дополнительной валидации.
- Эксперименты ограничены безопасным слоем descriptor/enumeration и не включают сценарии выполнения кода на хосте.
